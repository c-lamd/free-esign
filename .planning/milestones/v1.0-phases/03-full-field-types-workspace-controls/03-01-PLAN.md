<objective>
Extend the field data model and the PDF export engine to support all four new field types (initials, date, text, checkbox) and add the undo/redo history stack — the data and export spine that every Phase 3 field-type UI builds on.

This is the foundational slice: it migrates `placementMode: boolean` to `armedFieldType: FieldType | null`, widens `PlacedField` to a discriminated union with an optional `textValue`, adds the initials draw-modal state seam, adds the manual undo/redo snapshot stack, and extends `exportSignedPdf` to draw text/date as Helvetica `drawText` and checkbox as a bold ASCII `'X'` — all through the existing EXP-02 incremental-save path so the zero-alteration byte-identity guarantee holds for every new type.

Per CONTEXT.md Area 1 (field model extension), Area 3 (undo/redo mechanism), Area 5 (export of new field types), and the EXP-02 invariant.

Purpose: Without the union + export branches, no new field type can be stored or downloaded. This plan makes "store and export a text/date/checkbox field, byte-identical" provable at the data layer; Plan 02 makes it clickable.
Output: Updated `fieldStore.ts` and `exportPdf.ts`, plus extended `fieldStore.test.ts` and `exportPdf.test.ts` covering the new types, undo/redo, history cap, and EXP-02 for text/checkbox/date.
</objective>

| T-03-01 | Tampering | exportPdf.ts drawText(textValue) | accept | pdf-lib encodes textValue as WinAnsi PDF text operators (glyphs), not interpreted as PDF operators or HTML — no injection vector (RESEARCH Security Domain; ASSUMED A5, no CVE search) |
