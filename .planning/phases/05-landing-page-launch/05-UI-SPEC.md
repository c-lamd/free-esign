# Phase 5 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none — hand-crafted inline-style + Tailwind v4 components (same as Phase 1–4) |
| Icon library | none — inline SVG only (lock/shield for privacy section, numbered step markers for how-it-works) |
| Font | system-ui stack — `-apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif` (UI chrome); script fonts (Dancing Script, Great Vibes, Pacifico) available for hero decorative flourish only (see Typography) |

**Rationale:** Phase 5 continues the established hand-crafted pattern. No shadcn or new design system dependency is introduced. The landing is a new view, not a new tech surface — all existing tokens apply. Script fonts (already self-hosted via `@font-face` in `src/index.css`) may be used as a tasteful decorative element in the hero to signal what the tool does; they are scoped as decorative content, not a UI type-scale role.

**New dependencies:** None. No router, no analytics, no BMC script. Any icon used is inline SVG. PRV-03 is hard: zero new third-party requests.

## Spacing Scale

Inherited from Phase 1–4 unchanged. All tokens declared in `src/index.css` `:root` are reused exactly. Phase 5 introduces one landing-specific spacing value for the hero vertical breathing room and a max-width container — declared below as Phase-5 landing additions.

| Token | Value | Phase 5 Usage |
|-------|-------|--------------|
| xs | 4px | Step-number-to-text gap, footer icon gap |
| sm | 8px | CTA button inner padding (vertical), footer link padding |
| md | 16px | Section horizontal padding (mobile), card inner padding |
| lg | 24px | Between how-it-works steps, between hero elements, footer rhythm |
| xl | 32px | Section vertical gap between landing sections |
| 2xl | 48px | Hero top/bottom padding (mobile), section top/bottom padding (desktop) |
| 3xl | 64px | Hero top/bottom padding (desktop) |

**Phase-5 landing additions (not off-scale — declared here explicitly):**

| Token | Value | Usage | Justification |
|-------|-------|-------|---------------|
| `max-width-landing` | 720px | Max-width container for hero and all landing sections | Landing content reads better narrower than full-viewport; 720px is a comfortable reading width for the candid hero copy at body scale. Constraint prevents overly wide line lengths. |
| `hero-display` | 36px | Hero h1 only (see Typography table) | The hero headline needs more visual weight than the existing 24px Display to establish hierarchy on a marketing surface. 36px at weight 600 is proportional to the page without requiring a new weight. Declared as a landing-specific addition. |

**Exceptions:**
- CTA button: minimum 44px height (WCAG 2.5.5), minimum 200px width on mobile for comfortable tapping.
- Step number circles: 40px × 40px (nearest multiple of 4 above the 36px minimum; the 4px gap closes to the text at the declared xs).
- Footer: `padding: 32px 16px` (xl horizontal desktop, md mobile).

## Typography

All Phase 1–4 sizes and weights carry forward. Phase 5 adds one landing-only display size for the hero h1, declared explicitly below.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 16px | 400 (regular) | 1.5 | Hero candid copy paragraphs, how-it-works step descriptions, privacy explainer body, footer copy |
| Label | 14px | 400 (regular) | 1.4 | Footer link labels, step sub-labels, BMC link text |
| Heading | 20px | 600 (semibold) | 1.2 | Section headings ("How it works", "Your privacy, guaranteed"), footer section heading |
| Display | 24px | 600 (semibold) | 1.1 | "FreeESign" wordmark in landing header — unchanged from tool view |
| **Hero Display** *(Phase-5 landing addition)* | **36px** | **600 (semibold)** | **1.15** | **Hero h1 only** — "I built this because I couldn't find a PDF signer that was actually free." This is the only element using 36px; it does not propagate to other sections. |

**Font weights declared:** 400 and 600 only. No third weight added in Phase 5.

**Script font usage (decorative content, not a type-scale role):**

The hero section may include a single decorative signature flourish — e.g., the word "Free" or a stylised "Sign here" rendered in Dancing Script or Great Vibes — to visually communicate what the tool does before the user reads the body copy. Constraints:

- Scoped to one element, inline, not a heading or interactive element.
- CSS: `font-family: "Dancing Script", cursive; font-size: 28px; font-weight: 400; color: var(--color-accent); line-height: 1.2; user-select: none; aria-hidden: true`
- `aria-hidden="true"` — purely decorative, not part of the accessible text tree.
- Does NOT appear in the type-scale table because it is content decoration (like an image), not a UI role.
- Uses the already-self-hosted `@font-face` from `src/index.css` — zero new font requests.

## Color

All 8 existing CSS tokens from `src/index.css` are reused. No new CSS custom properties are added in Phase 5.

| Role | Value | CSS Token | Usage |
|------|-------|-----------|-------|
| Dominant (60%) | `#F9FAFB` (gray-50) | `--color-surface` | Landing page background, section alternating tint |
| Secondary (30%) | `#FFFFFF` (white) | `--color-surface-elevated` | Landing header bar, how-it-works cards, privacy section card, footer background |
| Canvas backdrop | `#E5E7EB` (gray-200) | `--color-canvas` | Not used in landing sections (tool-only token) |
| Accent (10%) | `#2563EB` (blue-600) | `--color-accent` | See extended reserved list below |
| Destructive | `#DC2626` (red-600) | `--color-destructive` | Not used in landing (no destructive actions) |
| Text primary | `#111827` (gray-900) | `--color-text-primary` | Hero h1, section headings, step body copy, footer headings |
| Text secondary | `#6B7280` (gray-500) | `--color-text-secondary` | Hero supporting paragraphs, step sub-labels, privacy body copy, footer copy |
| Border | `#D1D5DB` (gray-300) | `--color-border` | Step card borders, footer top border, header bottom border on landing view |

**60/30/10 discipline:** The landing page is predominantly neutral. Gray-50 (`--color-surface`) covers the page background (60%). White (`--color-surface-elevated`) covers the header, how-it-works step cards, and footer sections (30%). Accent is reserved exclusively for the primary CTA buttons and the decorative script flourish — nothing else in the landing uses blue.

**Accent (`#2563EB`) reserved for exactly (Phase 1–4 list extended with Phase-5 additions):**
1. "Browse files" button background (Phase 1)
2. Drag-over state border of the upload zone (Phase 1)
3. Prev/next navigation button focus ring (Phase 1)
4. Spinner stroke color (Phase 1)
5. "Add signature" button in draw-pad modal (Phase 2)
6. "Download PDF" button in TopBar (Phase 2)
7. Selected placed-field widget border (Phase 2)
8. Resize handle dots on a selected widget (Phase 2)
9. Focus rings on all interactive elements (Phase 2 pattern)
10. Active/armed field-palette button background (Phase 3)
11. Undo/redo button focus ring only (Phase 3)
12. Zoom +/− button hover icon (Phase 3)
13. **Landing primary CTA button background** — "Start signing" hero button and the bottom CTA repeat (Phase 5 addition)
14. **Decorative script flourish color** — the single hero signature decoration rendered in a script font (Phase 5 addition; aria-hidden)

Accent is NOT used for: the landing header background, section headings, body copy, step number circles, footer text, the BMC link text (ghost link style, `--color-text-secondary`), or any background fill on the landing except the primary CTA buttons.

**Step number circles:** `background: var(--color-surface); border: 2px solid var(--color-border); color: var(--color-text-primary); font-weight: 600; font-size: 16px` — neutral, not accent. Accent is not used for structural/decorative chrome, only interactive CTAs.

**Privacy section background:** Uses `--color-surface` (gray-50) tint as an alternating section to create visual separation without introducing a new color — same token, used as a section tint.

New components for Phase 5. All follow the Phase 1–4 pattern: inline-style objects with Tailwind v4 utility classes for layout; `var(--color-*)` tokens for all color references.

### LandingPage (new top-level component)

Rendered when `view === 'landing'` in Zustand. A single scrollable page composed of:
1. LandingHeader (minimal — wordmark only)
2. HeroSection
3. HowItWorksSection
4. PrivacySection
5. LandingFooter

**Overall layout:**
- `<main role="main">` wrapping HeroSection + HowItWorksSection + PrivacySection
- Each section is full-width; content constrained to `max-width: 720px; margin: 0 auto; padding: 0 16px` (the `max-width-landing` token)
- Background: `--color-surface` (gray-50)
- No TopBar from the signing tool — the LandingHeader replaces it on this view

### LandingHeader (new, landing-view only)

Replaces the signing-tool TopBar on the landing view. A minimal header bar.

**Layout:**
- Height: 56px fixed (matches TopBar height for visual consistency)
- `position: sticky; top: 0; z-index: 10`
- Background: `var(--color-surface-elevated)` (white)
- Bottom border: `1px solid var(--color-border)`
- Left slot: "FreeESign" wordmark — Display size (24px/600), `--color-text-primary`, not a link (user is already on landing)
- Right slot: "Sign a document" ghost link — Label size (14px/400), `--color-accent`, underline on hover; clicking transitions view → `'empty'`. This is a secondary entry point for users who scroll to the top and want to start.
- Horizontal padding: 16px (md), constrained within `max-width: 720px; margin: 0 auto`

**TopBar wordmark change (existing component):** When `view !== 'landing'` (i.e. the signing tool is active), the existing TopBar wordmark becomes a button that calls `goToLanding()` — `cursor: pointer`, `--color-text-primary` color unchanged, no underline. On hover: `--color-accent` color. `aria-label="FreeESign — return to home"`.

The primary landing section. Full-width, centered content column.

**Layout:**
- `padding: 64px 16px` desktop / `padding: 48px 16px` mobile (using 3xl/2xl tokens)
- `text-align: center`
- Content column: `max-width: 720px; margin: 0 auto`
- Background: `--color-surface` (gray-50)
- Stacking order (top to bottom):
  1. Decorative script flourish (optional, aria-hidden)
  2. Hero h1
  3. Hero subheading paragraph(s) — the candid first-person copy
  4. Primary CTA button
  5. Short reassurance line beneath the CTA

**Decorative script flourish:**
- `<span aria-hidden="true">` containing "FreeESign" or a signature-style word rendered in Dancing Script
- `font-family: "Dancing Script", cursive; font-size: 28px; color: var(--color-accent); line-height: 1.2; display: block; margin-bottom: 8px`
- Purely decorative — not in the tab order, not read by screen readers

**Hero h1:**
- Text: see Copywriting Contract
- `font-size: 36px; font-weight: 600; line-height: 1.15; color: var(--color-text-primary); margin-bottom: 24px`
- This is the only h1 on the page (landmark heading)

**Hero paragraphs (candid copy):**
- 1–3 short paragraphs, Body size (16px/400), `--color-text-secondary`, `line-height: 1.5`, `max-width: 600px; margin: 0 auto 24px`
- See Copywriting Contract for exact copy

**Primary CTA button:**
- Label: "Start signing — it's free"
- Style: `background: var(--color-accent); color: #FFFFFF; font-size: 16px; font-weight: 600; padding: 12px 32px; border-radius: 8px; border: none; cursor: pointer; min-height: 48px; min-width: 200px; transition: background 0.15s`
- Hover: `background: #1D4ED8` (blue-700)
- Active: `background: #1E40AF` (blue-800)
- Focus: `outline: 2px solid var(--color-accent); outline-offset: 2px`
- `aria-label="Start signing — opens the document uploader"`
- On click: `setView('empty')` (transitions out of landing into the signing tool)
- Note: uses 16px/600 instead of 14px/400 for the CTA to give it primary visual weight appropriate for a hero button. This is Body size at semibold — no new size token.

**Reassurance line beneath CTA:**
- Text: "No account needed. No uploads. Works in your browser."
- `font-size: 14px; font-weight: 400; color: var(--color-text-secondary); margin-top: 16px`

- If `BUY_ME_A_COFFEE_URL` contains `"PLACEHOLDER"`, the link is rendered but with a `<!-- TODO: replace BUY_ME_A_COFFEE_URL -->` comment; the executor should not hide it conditionally (the user can remove it manually post-deploy).

**State machine additions (Zustand extension):**

| State slice | Change | Notes |
|-------------|--------|-------|
| `ViewState` | Add `'landing'` to the union | `'landing' \| 'empty' \| 'loading' \| 'error' \| 'loaded'` |

## Accessibility Requirements

All Phase 1–4 requirements carry forward. Phase 5 landing additions:

- **Landmark regions:** The landing page uses semantic HTML landmarks — `<header>` (LandingHeader), `<main>` (hero + sections), `<footer>` (LandingFooter). Each section within `<main>` uses `<section aria-labelledby="{section-heading-id}">` so screen readers can navigate by section.
- **Heading hierarchy:** One `<h1>` per page (the hero headline). Section headings use `<h2>`. No h3 in this phase (step titles are visually heading-size but if they use `<h3>` that is fine; confirm they do not appear before the h2 in DOM order).
- **Hero h1 language:** The h1 is `lang` unset (inherits document language). Ensure `<html lang="en">` is set in `index.html`.
- **Script flourish accessibility:** `aria-hidden="true"` on the decorative script element — it is not part of the accessible text tree. Screen readers read the h1 directly.
- **CTA button vs link:** Both CTA elements ("Start signing — it's free" and the repeat "Start signing") are `<button>` elements (they trigger a Zustand state change within the same page, not navigation to a new URL). Accessible name is provided by visible label text. `aria-label` supplements with destination context (see Copywriting Contract).
- **LandingHeader "Sign a document":** This triggers `setView('empty')` — also a `<button>`, not an `<a>`, since no URL change occurs. Accessible name from visible text.
- **TopBar wordmark back-navigation:** A `<button>` wrapping the wordmark text. `aria-label="FreeESign — return to home"`. Visible text "FreeESign" already conveys the brand; aria-label adds the navigation intent.
- **BMC link:** The BMC `<a>` is a genuine outbound link (external URL, opens in new tab). It must include `rel="noopener noreferrer"` and the aria-label must disclose the new-tab behavior: `"Support FreeESign — Buy Me a Coffee (opens in new tab)"`.
- **Focus rings:** All interactive elements use the same 2px `--color-accent` outline, 2px offset pattern established in Phase 1–4.
- **Touch targets:** CTA buttons: `min-height: 48px` (above the 44px WCAG minimum). LandingHeader "Sign a document" link: `min-height: 44px; padding: 12px 0`. BMC link: `min-height: 44px` via padding.
- **Color contrast (new pairs):**
  - White text on `--color-accent` (#2563EB) — hero CTA button: 4.54:1 — passes AA (same as Phase 1/2 primary buttons)
  - `--color-text-primary` (#111827) on `--color-surface-elevated` (#FFFFFF) — hero h1: 16.75:1 — passes AAA
  - `--color-text-secondary` (#6B7280) on `--color-surface` (#F9FAFB) — privacy copy: 4.63:1 — passes AA
  - `--color-text-primary` (#111827) on `--color-surface` (#F9FAFB) — step titles: 16.47:1 — passes AAA
  - `--color-accent` (#2563EB) on `--color-surface` (#F9FAFB) — script flourish: 4.65:1 — passes AA (large text)
- **`aria-live` on view transitions:** When view changes from `'landing'` to `'empty'`, no announcement is needed — the DOM content change is immediately apparent and the UploadZone is fully interactive. No `aria-live` region required for the view machine itself.
- **No motion:** The only transition declared is `background 0.15s` on the CTA button hover — this is a color change, not motion, and does not require a `prefers-reduced-motion` guard. If any entrance animations are added, they must respect `prefers-reduced-motion: reduce`.

## PRV-03 Contract (Zero Third-Party Requests)

This is a non-visual hard constraint that the executor must enforce and test.

**What the landing page must NOT include:**

- No `<script src="...">` pointing to any external domain
- No `<link href="...">` to an external stylesheet or font CDN (Google Fonts, etc.)
- No `@font-face` with an external `src` URL — all fonts are already self-hosted under `public/fonts/`
- No `<img src="...">` referencing an external domain
- No `fetch(...)` or `XMLHttpRequest` targeting any external origin (existing fetch for fonts uses local `/fonts/` paths — unchanged)
- No analytics scripts (no Google Analytics, no Plausible, no Fathom)
- No error tracking (no Sentry, no Datadog)
- No chat widgets, no social embeds, no tracking pixels
- No Vercel Analytics or Vercel Speed Insights (these inject third-party scripts — must not be enabled)

**What IS permitted:**

- The BMC `<a href>` link is a navigation target that the user clicks intentionally. The browser only makes a request when the user clicks it. This is equivalent to any outbound hyperlink and is explicitly permitted.
- All npm packages bundled into `dist/` by Vite are same-origin at runtime.
- The pdfjs-dist worker at `public/pdf.worker.min.mjs` is a same-origin asset.
- Script fonts at `public/fonts/` are same-origin assets.

**Automated guard (required test):**

A Vitest test in `src/test/privacyGuard.test.ts` must:

1. Scan the built `dist/` directory (or fallback: the `src/` and `public/` source tree) for any occurrence of `https://` or `http://` appearing in:
   - `<script src=...>`, `<link href=...>` in HTML files
   - `@font-face { src: url(https://...` in CSS files
   - `fetch('https://` or `fetch("https://` in JS/TS files outside the font-loading allowlist
2. Assert that zero such patterns are found.
3. Allow exactly: the BMC `https://www.buymeacoffee.com/` anchor href, and any `https://` that appears only in comments or string literals that are provably anchor `href` attributes (not `fetch`, `src`, `href` on link/script tags).

Implementation note for the executor: a simple regex scan of `src/` and `index.html` is sufficient for v1. The full `dist/` scan after `npm run build` is the gold standard but requires the build to run first; the source scan is CI-friendly.
